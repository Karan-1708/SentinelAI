# Security Policy

SentinelAI is a portfolio / demo project, but I take responsible disclosure seriously and
treat every report on its merits. Thanks for taking the time to read.

## Reporting a vulnerability

Please email **karangill.1708@gmail.com** with the subject line **`[SentinelAI Security]`**.

Include as much of the following as you can:

- A concise description of the issue and the affected component (file path / route).
- Reproduction steps or a proof-of-concept — the smallest reliable case that exhibits the bug.
- The impact you observe or believe possible (what an attacker can do).
- Your preferred contact for follow-up and, if applicable, the name you'd like credited.

**Please do not** open a public GitHub issue for a suspected vulnerability, and please do not
tweet or blog about it before the disclosure timeline below has elapsed.

## Response SLA

- **Acknowledgement**: within 3 business days of receipt.
- **Initial triage** (severity assessment + reproduction confirmation): within 10 business days.
- **Fix or mitigation**: schedule depends on severity — see below.

I will keep you in the loop with progress updates at reasonable intervals.

## Severity and fix targets

| Severity | Examples | Target fix |
|----------|----------|------------|
| Critical | Auth bypass, unauthenticated RCE, JWT confusion / signature bypass, secret exfiltration | 7 days |
| High | Privilege escalation across roles, SQL / command / template injection, PDF template injection, path traversal, insecure deserialisation | 21 days |
| Medium | Broken access control on non-critical routes, missing rate limits enabling meaningful abuse, DoS via unbounded input, insecure cookie / header defaults | 60 days |
| Low | Information disclosure without impact, missing best-practice headers on non-critical paths | best-effort |

## Scope

**In scope**

- The `main` branch of this repository.
- The FastAPI backend (`api/`), the ML + explainability code (`models/`, `explainability/`,
  `ingestion/`), the React dashboard (`dashboard/`), and the CI workflow.
- Infrastructure examples under `docker/` insofar as the defaults are insecure.

**Out of scope**

- Findings that require compromising the host, hypervisor, or the developer's endpoint.
- DoS from unlimited request volume against a demo endpoint you host yourself.
- Self-XSS or issues that require the victim to paste attacker-controlled content into their
  own DevTools.
- Best-practice complaints without a concrete attack scenario (e.g. "you're not using WAF X").
- The dev-only htpasswd (`admin` / `dev`) shipped by `scripts/gen_dev_certs.sh` — it is
  documented as replace-before-deploy.
- Vulnerabilities in third-party services (Elasticsearch, MLflow, MinIO, Postgres) themselves —
  report those upstream. Misconfiguration of those services in this repo *is* in scope.
- Reports generated by unauthenticated automated scanners without validation.

## Vulnerability classes I am particularly interested in

- Auth bypass or role-escalation across `viewer` / `analyst` / `admin`.
- JWT signature or claim-confusion issues (`iss`, `aud`, `exp`, algorithm downgrade).
- Injection: SQL, log, PDF / Platypus template, prompt injection, CRLF.
- Broken access control on `/incidents/{id}/status` and `/reports/{id}`.
- RCE via model artifact deserialisation (`joblib.load`, `xgb.Booster.load_model`).
- Path traversal, zip-slip, or SSRF via the CICIDS downloader or the MITRE mapper.
- Secrets exposure in logs, error responses, `docker inspect`, or the client bundle.
- WebSocket authentication bypass or Cross-Site WebSocket Hijacking.
- XSS in the dashboard — including via server-supplied data that bypasses zod.

## Safe harbour

I authorise good-faith security research consistent with this policy. If you follow it, I will
not pursue or support any legal action against you (CFAA, DMCA, or analogous). You are still
responsible for complying with applicable law.

Please:

- Access only data that belongs to you or that you have explicit permission to access.
- Do not degrade the service for other users.
- Do not exfiltrate more data than necessary to prove impact.
- Give me reasonable time to fix an issue before public disclosure.

## Coordinated disclosure

I will publicly disclose the fix — with credit if you'd like — **90 days** after a fix is
shipped, or **120 days** after the initial report, whichever comes first. If a fix is not
possible within that window, I will contact you to agree an extension.

## Known limitations

- This is a personal / portfolio project. There is no bug bounty program and no monetary
  reward — only credit and, at your option, a public mention.
- I am one person. Response times reflect real-world availability.
